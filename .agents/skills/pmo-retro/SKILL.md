# pmo-retro — 复盘助手

> 在里程碑完成或项目阶段结束时，自动聚合会议纪要、待办完成/过期数据，AI 辅助生成结构化复盘报告。

## 执行方式

```bash
# 交互式复盘（选择范围）
claude pmo-retro

# 基于里程碑范围复盘
claude pmo-retro --milestone MILE-001

# 指定起始日期到今天
claude pmo-retro --since 2026-06-01

# 预览模式，不归档
claude pmo-retro --dry-run

# 指定项目（无需切换上下文）
claude pmo-retro --project XRay
```

## 前置条件

已通过 `pmo-use` 设置当前项目（或使用 `--project` 指定）。

复盘质量依赖数据积累量：建议至少有 2 次会议记录 + 10 条已完成待办才能生成有价值的复盘报告。

**所有飞书 API 写操作遵循公共错误重试策略（见 CLAUDE.md）：3 次指数退避重试（1s/3s/5s）。**

## 执行流程

### 公共：待处理队列检查

> 📋 详见 [`_shared/pending-queue-check.md`](../_shared/pending-queue-check.md)。

### 配置加载

1. 按 CLAUDE.md「读取当前项目配置」规则加载项目配置
2. 检查 `schemaVersion`，执行必要的版本迁移
3. 执行配置完整性校验（必填字段：`project.name`、`larkResources.baseAppToken`、`larkResources.baseTableIds.*`、`larkResources.wikiSpaceId`）

### 第1步：确定复盘范围

| 参数 | 范围 |
|------|------|
| 默认（无参数）| 进入交互选择（见下方） |
| `--milestone <ID>` | 里程碑计划开始日 ~ 实际完成日（或 today） |
| `--since YYYY-MM-DD` | 指定日期 ~ today（最长不超过 90 天） |

**交互式范围选择（无参数时）：**

```
请选择复盘范围：
  1. 最近 30 天（{startDate} ~ {today}）
  2. 最近 60 天
  3. 基于里程碑
  4. 自定义起始日期
请输入序号：
```

选择「基于里程碑」后，展示已完成或进行中的里程碑列表供选择：
```
请选择里程碑：
  1. [MILE-001] Beta版发布（06-01 ~ 06-15）✅ 已完成
  2. [MILE-002] UAT评审（06-08 ~ 06-20）🔄 进行中
请输入序号：
```

### 第2步：并行拉取三类数据

对确定的时间范围，**并行**查询：

**① 会议纪要文档（通过 lark-wiki + lark-doc）：**
- 通过 `lark-wiki` 列出知识库 `01-会议纪要/` 目录下的文档节点
- 按文档名过滤出日期在范围内的文档（格式：`YYYYMMDD-*`）
- 通过 `lark-doc` 逐个读取文档正文（最多读取 10 篇，优先最新）
- 提取内容：关键决策段落、讨论要点摘要
- 加载失败时（无 wiki 节点/无文档）：标注「未找到会议纪要文档，复盘将基于 Base 数据」，继续执行

**② 待办数据（通过 lark-base）：**
- 已完成待办（完成日期在范围内）：内容、负责人、完成日期、优先级
- 过期未完成待办（截止日期在范围内但状态≠已完成/已取消）：内容、负责人、截止日期、逾期天数
- 已取消待办（创建时间在范围内）：内容（了解哪些事项被放弃）

**③ 里程碑数据（通过 lark-base）：**
- 范围内状态发生变更的里程碑记录
- 计划日期 vs 实际完成日期（推算是否延期）

**数据量提示：**
```
📊 已加载：会议纪要 {N} 篇 · 完成待办 {N} 条 · 过期待办 {N} 条 · 里程碑 {N} 个
```
若完成待办 < 5 条，提示「⚠️ 数据量较少，复盘内容可能不够充分」。

### 第3步：AI 综合分析生成复盘报告

将以上三类数据注入 AI，生成以下结构的复盘报告：

**① 做得好的（What Went Well）**
从以下角度提取亮点：
- 完成质量高（P0/P1 待办准时完成率）
- 里程碑如期完成
- 会议纪要中重复出现的正面评价关键词（顺利、完成、达标、通过）
- 团队协作亮点（某成员高完成率）

**② 待改进的（What Could Be Better）**
从以下角度提取问题：
- 过期待办的规律性（同一负责人/同一来源/同一类型反复出现）
- 里程碑延期原因（从纪要中提取阻塞点）
- 会议纪要中重复出现的问题关键词（延期、阻塞、依赖、变更、未完成）
- 已取消待办的归因（需求变更 vs 优先级降低 vs 资源不足）

**③ 行动计划（Action Items）**
基于「待改进的」自动生成下阶段改进措施，每条行动计划包含：
- 改进内容（具体可执行的措施）
- 建议负责人（可选）
- 建议截止日期（可选）

**生成约束：**
- 每个分类 3-8 条，避免过多泛化描述
- 每条不超过 50 字，简洁直接
- 行动计划必须是可执行的具体措施，不输出「需要改进沟通」这类模糊建议

### 第4步：展示草稿供用户确认

```
📋 复盘报告草稿 · {项目名}（{startDate} ~ {endDate}）
════════════════════════════════════════

## ✅ 做得好的
1. {亮点1}
2. {亮点2}
...

## 🔧 待改进的
1. {问题1}
2. {问题2}
...

## 📌 行动计划
1. {措施1} — @{负责人建议}（{截止日期建议}）
2. {措施2} — @{负责人建议}
...

────────────────────────────────────────
数据来源：{N} 篇会议纪要 · {N} 条完成待办 · {N} 条过期待办

[确认归档] [修改] [确认并写入行动计划到Base] [仅导出] [取消]
```

| 选项 | 行为 |
|------|------|
| **确认归档** | 创建飞书文档 + 归档到 `05-项目资料/` |
| **修改** | 用户输入修改意见，AI 调整后重新展示 |
| **确认并写入行动计划到 Base** | 归档文档 + 将行动计划中的待办写入 Base 待办表 |
| **仅导出** | 生成本地 Markdown，不归档 |
| **取消** | 丢弃草稿 |

### 第5步：归档与可选写入待办

**归档复盘文档：**

1. 使用以下模板生成飞书文档正文：

```markdown
# {项目名} 复盘报告

> 复盘范围：{startDate} ~ {endDate}
> 生成时间：{today}

## ✅ 做得好的
{内容}

## 🔧 待改进的
{内容}

## 📌 行动计划
{内容}

---

## 📊 数据摘要

| 维度 | 数据 |
|------|------|
| 完成待办 | {N} 条 |
| 过期待办 | {N} 条（完成率 {pct}%）|
| 里程碑完成 | {N}/{total} 个 |
| 覆盖会议 | {N} 次 |
```

2. 通过 `lark-doc` 创建飞书文档
3. 通过 `lark-wiki` 归档到 `05-项目资料/`
   - 文档标题格式：`{YYYYMMDD}-复盘-{里程碑名称或"近{N}天"}`
   - 命名冲突时追加序号

**写入行动计划到 Base（选「确认并写入行动计划」时）：**

对行动计划中每条措施，通过 `lark-base` 写入待办表：
- 内容 = 行动计划措施内容
- 负责人 = 按成员名称解析规则（见 CLAUDE.md）
- 截止日期 = 建议截止日期（按 `_shared/date-calc-rules.md` 计算，若无建议则留空）
- 来源 = `"复盘行动计划:{复盘文档标题}"`
- 状态 = 待处理
- 优先级 = P1-重要（默认）

写入前展示待写入待办预览，用户可调整：
```
📋 即将写入行动计划（{N} 条）：
  1. {措施内容} | @{负责人} | 截止: {日期}
  2. {措施内容} | @{负责人} | 截止: 待确认
确认写入？ [是] [否]
```

**输出确认：**
```
✅ 复盘报告已生成
══════════════════════════════
  文档：{YYYYMMDD}-复盘-{标识}
  归档：05-项目资料/
  链接：{wiki文档链接}
  行动计划：{N} 条已写入 Base（或：未写入）
```

### --dry-run 预览模式

只执行步骤 1-3，展示完整草稿后结束，**不创建文档、不归档、不写入 Base**。

底部显示：
```
── 预览模式，未写入 ──
使用 pmo-retro 正式生成
```

## 异常处理

| 场景 | 处理方式 |
|------|------|
| 无当前项目且未指定 --project | 提示「请先执行 pmo-use <项目名>」 |
| 找不到里程碑 ID | 提示「找不到里程碑 {ID}，通过 pmo-milestone 查看」 |
| 范围内无任何数据 | 提示「指定范围内无可用数据，建议扩大时间范围」，不继续 |
| 数据量过少（完成待办 < 5 条）| 展示警告但继续，AI 降低置信度措辞 |
| Wiki 文档读取失败 | 跳过该文档，在提示中标注；不影响主流程 |
| Wiki 归档失败（重试耗尽）| 提示 `❌ 归档失败` + `→ 请执行 pmo-archive <导出文件> 手动归档`（见 CLAUDE.md 人工介入出口） |
| 行动计划写入 Base 部分失败 | 记录失败条目，提示用户手动补录；成功的照常写入 |
| --since 超过 90 天 | 提示「建议复盘范围不超过 90 天，已自动截取最近 90 天」 |

## 边缘情况

| 场景 | 处理方式 |
|------|------|
| 无会议纪要文档 | 基于纯 Base 数据生成复盘，提示「无会议纪要，报告基于待办/里程碑数据」 |
| 所有待办均已完成（过期 0 条）| 「待改进的」显示「本阶段执行质量优秀，暂无明显问题」 |
| 行动计划中负责人未匹配 | 负责人留空，标注 ⚠️，不阻塞写入 |
