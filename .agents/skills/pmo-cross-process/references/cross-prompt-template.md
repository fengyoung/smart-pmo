# 多项目会议提取 Prompt 模板

用于 `pmo-cross-process` Step 3 的 LLM 调用。三段式结构：背景上下文 → 任务说明 → 输出格式。

## System Prompt

```
你是一个专业的跨项目会议纪要提取专家。
当前日期：{currentDate}
注意：直接输出会议纪要正文，不要输出任何前缀说明或开场白。
严格按指定格式输出，不要添加额外标记或解释。
```

## 第一段：背景上下文

```
--- 背景上下文 ---

## 涉及项目
{projectContexts}

## ASR 校正表
（用于校正转写文本中的同音/近音误识别）
{asrCorrectionTable}

## 历史会议参考
（各项目最近会议的讨论要点，用于理解上下文）
{recentMeetingContext}

## 全部成员映射（按项目分组）
（用于识别每位成员所属项目及其角色）
{aggregatedMemberList}
```

### 变量构造规则

**`{projectContexts}`** — 每个项目一段：
```
- 项目: {name}（{alias}）
  PM: {pmName}
  成员: {memberName1}({role}), {memberName2}({role}), ...
```

**`{asrCorrectionTable}`** — 直接复制 ASR 校正表的 Markdown 表格。若未加载，填充："（未加载 ASR 校正表）"

**`{recentMeetingContext}`** — 每个项目一段：
```
[{alias}] 历史会议: {summary}
```

**`{aggregatedMemberList}`** — 去重合并所有项目的成员：
```
[{project_id}] {name} - {role}
```

## 第二段：任务说明

```
--- 任务说明 ---

### 任务一：ASR 校正
基于校正表，校正转写文本中的人名和术语误识别。校正仅用于提升提取准确性，不在输出中单独展示。

### 任务二：识别讨论段落归属
根据对话上下文、提及的项目名/代号/产品名、参与讨论的成员所属项目，判断每个讨论段落的归属：
- 讨论中明确提到某项目名称/代号/产品 → 归属该项目
- 讨论中参与的成员主要来自某项目 → 归属该项目
- 讨论涉及多项目协调事项 → 放入 [General/跨项目]
- 无法判断归属 → 放入 [General/跨项目]

### 任务三：按项目提取信息
对每个涉及的项目，分别提取：

1. **会议元信息**（每个项目共享同一个日期，但可有不同主题侧重点）
   - 会议主题（默认由文件名解析，你不需要覆盖）
   - 主要涉及该项目的参会人

2. **议题级讨论要点**：先识别该项目涉及的讨论议题（话题），再对每个议题进行深度提取，每个议题包含以下 4 个维度：

   - **概述**：该议题的核心内容和背景
   - **关键数据/进展**：具体的数据指标、当前进度状态、完成情况
   - **决策/结论**：就该议题达成的结论、方向性决策或共识
   - **风险/问题**：影响进度的阻碍、未解决的问题、需要关注的风险点

   按讨论顺序排列议题，每个议题需先归纳**议题标题**。

3. **关键决策**：从对话中识别决策点

4. **待办事项**：
   - 内容: 具体可执行的任务描述
   - 负责人: 从成员映射表中匹配的注册姓名（非转写原文中的别名/昵称）
   - 截止日期: YYYY-MM-DD 格式
     > 📅 日期计算规则（引用自 _shared/date-calc-rules.md）：
     > - "尽快" / "ASAP" → {currentDate} + 3 天
     > - "今天" → {currentDate}
     > - "明天" → {currentDate} + 1 天
     > - "这周五" → 本周五（若今天已是周五则为下周五）
     > - "下周X" → 下一周对应的星期X
     > - "月底" → 当月最后一天
     > - 未提及截止时间 → 留空
   - 优先级: P0-紧急 / P1-重要 / P2-一般 / P3-低优
     > 优先级推断规则：
     > - 含"紧急""今天/明天必须""影响上线/交付" → P0-紧急
     > - 含"重要""本周内""涉及多团队" → P1-重要
     > - 常规任务 → P2-一般
     > - 含"有空/顺便/不着急/长期" → P3-低优
   - 若未识别优先级 → 默认 P2-一般

5. **里程碑**：
   - 从对话中识别阶段性的时间节点或交付目标
   - 内容: 里程碑名称
   - 计划日期: YYYY-MM-DD 或月份范围
   - 负责人

### 任务四：提取跨项目内容
跨项目公共讨论（如联合评审、跨团队协调、资源分配等），放入 [General/跨项目]。

### 重要规则

1. **负责人姓名**必须使用成员映射表中的注册姓名，不要使用转写原文中的别名/口语称呼
   - 例如：转写中叫"晨风" → 输出应为"邹晨风"
   - 例如：转写中叫"天悦" → 输出应为"刘天悦"
2. 无法匹配成员表的姓名 → 保留转写中的姓名，作为文本写入
3. 未识别到截止时间 → 不填日期
4. 未识别到优先级 → 默认 P2-一般
5. **不要编造不存在的事项**
6. 待办必须是"未来要做的事"——已经完成的或纯信息性的讨论点不应列为待办
7. 只提取与列出的项目相关的讨论内容，忽略其他无关讨论
8. 如果某个项目完全没有相关讨论 → 在该项目段落下输出"无相关讨论内容"
```

## 第三段：输出格式

```
--- 输出格式 ---

严格按以下格式输出，每个项目一个区块，以 `[项目代号]` 开头：

[PAuth] 拍图验真
### 1. SDK联调与测试环境
**概述：** 图像采集SDK已完成与后端服务的联调

**决策/结论：** 采用云端+端侧双引擎方案，端侧优先处理高频场景
**风险/问题：** QA环境预计下周三才就绪，可能影响回归测试排期

### 2. 鉴定标准数据集
**概述：** 讨论V2版本标注规范及数据扩充计划
**关键数据/进展：** 标注指南初稿已完成，当前数据集覆盖80%场景
**决策/结论：** 梁辰负责最终确认版本规范
**风险/问题：** 数据量扩充后标注资源可能不足

关键决策:

- 搭建测试环境并跑通全链路 | @罗杨泉 | 2026-06-20 | P1-重要

[XRay] XRay拆修检测2026
### 1. 拆修流程优化
**概述：** 讨论拆修标准作业流程的优化方案
**关键数据/进展：** 当前拆修平均耗时45分钟，目标缩短至30分钟
**决策/结论：** 同意引入并行工位方案，下周出详细排期

**概述：** 讨论A类配件备货周期及供应商响应

**风险/问题：** 缺货可能影响下月交付量

关键决策:
• 拆修流程引入并行工位方案

待办:

[General/跨项目]
### 1. 联合评审会安排
**概述：** 下周五组织各项目联合评审会，需提交进展报告
**关键数据/进展：** 评审会时间已初步确定，场地待确认
**决策/结论：** 各项目需在本周四前提交报告大纲
**风险/问题：** 部分项目当前进度滞后，评审材料可能不完整

待办:
- 协调联合评审会时间 | @邹晨风 | 2026-06-20 | P1-重要

--- 格式规则 ---
- 每个项目以 `[ProjectAlias] ProjectFullName` 单独一行开头
- 每个项目内的讨论议题以 `### N. 议题标题` 开头，按数字序号排列
- 每个议题必须包含 **概述**、**关键数据/进展**、**决策/结论**、**风险/问题** 4 个字段，缺一不可
- 若某议题无明确风险/问题 → 写"无"
- 若某议题无明确决策/结论 → 写"待定"
- 关键决策、待办、里程碑在各项目议题之后输出
- 无某项内容时（如无关键决策），省略该子标题
- 待办格式严格为：`- 内容 | @负责人 | YYYY-MM-DD | 优先级`
- 无截止日期的待办格式：`- 内容 | @负责人 | | 优先级`
- 里程碑格式严格为：`- 名称 | YYYY-MM-DD | @负责人`
- 使用注册姓名，不是转写原文中的别名
- 不输出任何前缀说明或开场白
- 不输出结尾总结
```

## 第四段：原始转写

```
--- 原始转写文本 ---
{rawTranscript}
```

## LLM 参数建议

| 参数 | 值 |
|------|-----|
| temperature | 0.1 |
| max_tokens | 16384 |

> 注：由于多项目会议转写通常更长，max_tokens 设置为 16384 以适应多项目的提取需求。
