# 多项目会议提取 Prompt 模板

用于 `pmo-cross-process` Step 3 的 LLM 调用。三段式结构：背景上下文 → 任务说明 → 输出格式。

## System Prompt

```
你是一个专业的跨项目会议纪要提取专家。
当前日期：{currentDate}
注意：直接输出会议纪要正文，不要输出任何前缀说明或开场白。
严格按指定格式输出，不要添加额外标记或解释。
```

## 第一段：背景上下文

```
--- 背景上下文 ---

## 涉及项目
{projectContexts}

## ASR 校正表
（用于校正转写文本中的同音/近音误识别）
{asrCorrectionTable}

## 历史会议参考
（各项目最近会议的讨论要点，用于理解上下文）
{recentMeetingContext}

## 全部成员映射（按项目分组）
（用于识别每位成员所属项目及其角色）
{aggregatedMemberList}
```

### 变量构造规则

**`{projectContexts}`** — 每个项目一段：
```
- 项目: {name}（{alias}）
  PM: {pmName}
  成员: {memberName1}({role}), {memberName2}({role}), ...
```

**`{asrCorrectionTable}`** — 直接复制 ASR 校正表的 Markdown 表格。若未加载，填充："（未加载 ASR 校正表）"

**`{recentMeetingContext}`** — 每个项目一段：
```
[{alias}] 历史会议: {summary}
```

**`{aggregatedMemberList}`** — 去重合并所有项目的成员：
```
[{project_id}] {name} - {role}
```

## 第二段：任务说明

```
--- 任务说明 ---

### 任务一：ASR 校正
基于校正表，校正转写文本中的人名和术语误识别。校正仅用于提升提取准确性，不在输出中单独展示。

### 任务二：识别讨论段落归属
根据对话上下文、提及的项目名/代号/产品名、参与讨论的成员所属项目，判断每个讨论段落的归属：
- 讨论中明确提到某项目名称/代号/产品 → 归属该项目
- 讨论中参与的成员主要来自某项目 → 归属该项目
- 讨论涉及多项目协调事项 → 放入 [General/跨项目]
- 无法判断归属 → 放入 [General/跨项目]

### 任务三：按项目提取信息
对每个涉及的项目，分别提取：

1. **会议元信息**（每个项目共享同一个日期，但可有不同主题侧重点）
   - 会议主题（默认由文件名解析，你不需要覆盖）
   - 主要涉及该项目的参会人

2. **议题级讨论要点**：先识别该项目涉及的讨论议题（话题），再对每个议题进行深度提取。**每个议题的 4 个维度必须展开到足以体现完整的讨论脉络，不能只写摘要性只言片语。** 每个维度包含以下内容层次：

   - **概述**（2-3段）：该议题的核心内容和完整背景
     - 为什么会讨论这个议题？涉及哪些人？前因后果是什么？
     - 讨论的叙事脉络——谁发起、谁回应、争议点在哪里、如何收束
     - 具体业务场景和上下文，让未参会的人也能读明白
   - **关键数据/进展**（2-3段）：涵盖所有具体的数据和事实
     - 数据指标：数量、比例、时长、金额等，须有具体数字（如"约1000+条/天"）
     - 时间节点：已完成的日期、计划中的截止日期
     - 状态描述：已跑通/已上线/调研中/待定等精确状态
     - 人员配置：谁负责什么、投入了多少资源
   - **决策/结论**（1-2段）：完整的决策链路
     - 谁提出了什么方案、有什么不同意见、争议点是什么
     - 最终决定是什么、依据是什么
     - 如果决策待定，写明后续的决策机制和责任人
   - **风险/问题**（1-2段）：具体到可执行的层面
     - 问题的具体表现是什么（不只是说"有风险"，要说清楚风险场景）
     - 影响范围有多大、会波及哪些环节
     - 有什么应对计划或缓解措施，谁在跟进

   按讨论顺序排列议题，每个议题需先归纳**议题标题**。

3. **关键决策**：从对话中识别决策点

4. **待办事项**：
   - 内容: 具体可执行的任务描述
   - 负责人: 从成员映射表中匹配的注册姓名（非转写原文中的别名/昵称）
   - 截止日期: YYYY-MM-DD 格式
     > 📅 日期计算规则（以下为 LLM 用快照，规范定义见 [`_shared/date-calc-rules.md`](../_shared/date-calc-rules.md)）：
     > - "尽快" / "ASAP" → {currentDate} + 3 天
     > - "今天" → {currentDate}
     > - "明天" → {currentDate} + 1 天
     > - "这周五" → 本周五（若今天已是周五则为下周五）
     > - "下周X" → 下一周对应的星期X
     > - "月底" → 当月最后一天
     > - 未提及截止时间 → 留空
   - 优先级: P0-紧急 / P1-重要 / P2-一般 / P3-低优
     > 优先级推断规则（以下为 LLM 用快照，规范定义见 [`_shared/date-calc-rules.md`](../_shared/date-calc-rules.md)）：
     > - 含"紧急""今天/明天必须""影响上线/交付" → P0-紧急
     > - 含"重要""本周内""涉及多团队" → P1-重要
     > - 常规任务 → P2-一般
     > - 含"有空/顺便/不着急/长期" → P3-低优
   - 若未识别优先级 → 默认 P2-一般

5. **里程碑**：
   - 从对话中识别阶段性的时间节点或交付目标
   - 内容: 里程碑名称
   - 计划日期: YYYY-MM-DD 或月份范围
   - 负责人

### 任务四：检测 `--projects` 之外的已知项目
检查转写内容中是否提到了 `--projects` 列表之外的已知项目信息。已知项目列表（含名称、代号）已在背景上下文的「涉及项目」中给出。

识别规则：
- 转写中明确提到某项目名称/代号（如"拍照3.0"对应 PIC-3.0）
- 转写中讨论的内容与某项目的业务领域高度相关
- 讨论的参与者主要来自某已知项目团队

若检测到 → 在输出中以 `[PIC-3.0] ⚠️ 未在 --projects 中指定` 格式输出，内容提取方式与正式项目一致（议题级×4维度）。同时在待办列表末尾追加一行 `⚠️ 检测到额外项目：{项目名}，如需写入请在下次执行时添加 --projects`

### 任务五：提取跨项目内容
跨项目公共讨论（如联合评审、跨团队协调、资源分配等），放入 [General/跨项目]。

### 重要规则

1. **负责人姓名**必须使用成员映射表中的注册姓名，不要使用转写原文中的别名/口语称呼
   - 例如：转写中叫"晨风" → 输出应为"邹晨风"
   - 例如：转写中叫"天悦" → 输出应为"刘天悦"
2. 无法匹配成员表的姓名 → 保留转写中的姓名，作为文本写入
3. 未识别到截止时间 → 不填日期
4. 未识别到优先级 → 默认 P2-一般
5. **不要编造不存在的事项**
6. 待办必须是"未来要做的事"——已经完成的或纯信息性的讨论点不应列为待办
7. 只提取与列出的项目相关的讨论内容，忽略其他无关讨论
8. 如果某个项目完全没有相关讨论 → 在该项目段落下输出"无相关讨论内容"
9. **ASR 校正反馈**：提取过程中，如果发现转写中的人名/术语疑似 ASR 误识别且未在 ASR 校正表中覆盖，在待办列表末尾输出候选修正项。格式：
   ```
   ⚠️ 建议添加到 ASR 校正表：{误识别写法} → {正确写法}
   ```
   候选条件：无法在成员映射表中匹配到，但通过上下文可推断出正确人名的口语称呼/别名。纯业务术语不在此范围。
```

## 第三段：输出格式

```
--- 输出格式 ---

严格按以下格式输出，每个项目一个区块，以 `[项目代号]` 开头：

[PAuth] 拍图验真
### 1. SDK联调与测试环境
**概述：** 图像采集SDK已完成与后端服务的联调。本次讨论聚焦在SDK接入后的测试方案和部署策略。罗杨泉汇报了联调进展——后端API接口已全部对接完成，目前正在进行异常场景的边界测试。刘天悦提出需要评估端侧处理与云端处理的成本差异，建议高频场景优先走端侧以减少服务器压力。后端团队反馈端侧模型的裁剪工作已完成，单帧处理耗时从45ms优化到22ms，满足实时性要求。会议后半段讨论了QA测试环境的部署计划，目前依赖的GPU服务器由于采购流程尚未完成，环境搭建进度存在不确定性。

**关键数据/进展：**
- 接口联调通过率100%，全量接口（27个）已通
- 端侧模型裁剪完成：单帧处理 45ms → 22ms，降低51%
- QA测试环境依赖GPU服务器采购，流程尚未走完
- 已完成的联调工作覆盖了图像上传、预处理、推理、结果回传全链路
- 异常场景测试进度：已完成80%（超时处理、网络断开、图片损坏等场景）

**决策/结论：**
经过讨论，会议确认采用云端+端侧双引擎方案——端侧优先处理高频场景（如基础信息校验、型号识别），云端处理复杂场景（如细粒度瑕疵检测和需要大模型的判定）。刘天悦提出此方案可以平衡实时性和准确率，同时也为后续离线场景留出扩展空间。梁辰建议端侧模型以周为迭代周期，云端以双周为周期，罗杨泉表示后端可以支持这个节奏。

**风险/问题：**
QA测试环境预计下周三才就绪，比预期晚了一周，主要瓶颈是GPU服务器的采购审批流程。这可能导致回归测试排期紧张——如果本周五前环境仍不能搭建，原定下周一的集成测试可能需要延期。罗杨泉建议先使用开发环境做部分测试，但开发环境没有GPU加速，推理耗时是生产环境的3-5倍，只能覆盖接口层面的测试。刘天悦表示可以协调采购部门加速流程，预计明天给出明确时间表。

### 2. 鉴定标准数据集
**概述：** 讨论了V2版本标注规范的修订和数据扩充计划。标注团队反馈当前数据集在部分偏类场景（曲面屏、暗光环境）覆盖不足，影响了模型的泛化能力。梁辰展示了最新的标注质量分析报告——整体标注准确率98.2%，但偏类场景的标注一致性仅为82%。团队决定针对这些场景做一轮重新标注规范培训，同时扩充数据集以提升覆盖。数据扩充的来源包括历史工单图片和线下仓库实地采集。

**关键数据/进展：**
- 标注指南初稿已完成，覆盖80%的检测场景
- 偏类场景标注一致性仅82%，远低于95%的目标线
- 当前数据集总量：12万张标注图片
- 数据扩充目标：计划扩充至20万张，覆盖95%场景
- 标注团队当前产能：约800张/天，完成扩充预计需100个工作日

**决策/结论：**
标注规范V2版本由梁辰负责终版确认，王柳坤负责标注团队的培训对齐。数据扩充采用双源策略：①从历史工单中筛选可用的退换货图片（预估可获取5万张）；②线下仓库实地采集（预估可获取3万张，但需要协调仓库时间和场地）。梁辰建议优先处理历史工单数据，因为成本更低且无需外勤。郭煜彬建议标注规范定稿后本周五前完成培训，下周一开始新批次标注。

**风险/问题：**
数据量扩充后现有标注资源可能不足——当前标注团队只有5人，如果扩大到20万张，即使加上外部标注资源也需要2-3个月。此外，历史工单图片的质量参差不齐，部分图片存在过曝、遮挡等问题，筛选清洗环节需要额外投入。刘天悦建议考虑引入主动学习策略，让模型先筛选出不确定的样本优先标注，以提升标注资源的投入产出比。

关键决策:
• 云端+端侧双引擎方案确认——端侧负责高频场景，云端负责复杂场景
• 端侧模型以周为迭代周期，云端以双周为周期
• 标注规范V2版本：梁辰终版确认，王柳坤负责培训对齐
• 数据扩充采用双源策略（历史工单+仓库采集）

待办:

- 完成标注团队V2规范培训 | @王柳坤 | 2026-06-27 | P1-重要

--- 格式规则 ---
- 每个项目以 `[ProjectAlias] ProjectFullName` 单独一行开头
- 每个项目内的讨论议题以 `### N. 议题标题` 开头，按数字序号排列
- 每个议题必须包含 **概述**、**关键数据/进展**、**决策/结论**、**风险/问题** 4 个字段，缺一不可
- 若某议题无明确风险/问题 → 写"无"
- 若某议题无明确决策/结论 → 写"待定"
- 关键决策、待办、里程碑在各项目议题之后输出
- 无某项内容时（如无关键决策），省略该子标题
- 待办格式严格为：`- 内容 | @负责人 | YYYY-MM-DD | 优先级`
- 无截止日期的待办格式：`- 内容 | @负责人 | | 优先级`
- 里程碑格式严格为：`- 名称 | YYYY-MM-DD | @负责人`
- 使用注册姓名，不是转写原文中的别名
- 不输出任何前缀说明或开场白
- 不输出结尾总结
```

## 第四段：原始转写

```
--- 原始转写文本 ---
{rawTranscript}
```

## LLM 参数建议

| 参数 | 值 |
|------|-----|
| temperature | 0.1 |
| max_tokens | 16384 |

> 注：由于多项目会议转写通常更长，max_tokens 设置为 16384 以适应多项目的提取需求。
