# pmo-import — 批量导入

> 将 CSV / Excel 文件中的待办和里程碑批量导入 Base，适用于从其他系统迁移数据或批量补录。

## 执行方式

```bash
# 导入待办（交互式字段映射）
claude pmo-import --file <路径> --table todos

# 导入里程碑
claude pmo-import --file <路径> --table milestones

# 自动映射（文件名需匹配标准字段名）
claude pmo-import --file <路径> --table todos --auto-map

# 预览模式（不实际写入，仅验证）
claude pmo-import --file <路径> --table todos --dry-run

# 跳过重复检查（强制写入）
claude pmo-import --file <路径> --table todos --force
```

## 前置条件

已通过 `pmo-use` 设置当前项目。

**公共模式引用：**
- 📋 待处理队列检查详见 [`_shared/pending-queue-check.md`](../_shared/pending-queue-check.md)
- ⏱️ Base 操作超时遵循公共配置（见 CLAUDE.md）
- 🔄 Base 写入失败遵循公共错误重试策略：3 次指数退避（1s/3s/5s）
- 🔍 配置加载时执行 schemaVersion 检查和配置完整性校验（见 CLAUDE.md）
- 👤 负责人字段写入格式遵循公共规范（见 CLAUDE.md「公共：Base 写入负责人字段格式」）

## 支持的文件格式

| 格式 | 要求 |
|------|------|
| CSV | UTF-8 编码，首行为字段名 |
| XLSX | 第一个 Sheet，首行为字段名 |
| Markdown 表格 | 标准 `| col | col |` 格式 |

## 执行流程

### 第1步：解析文件

1. 自动检测文件格式（按扩展名 + 内容嗅探）
2. 解析首行为字段名列表
3. 验证至少有一行数据
4. 行数 > 200 时分批处理，每批 200 条

### 第2步：字段映射

**自动映射（`--auto-map`）：**
系统尝试将文件列名匹配到 Base 字段名（支持中英文模糊匹配）：

| 文件列名示例 | 映射到 Base 字段 |
|-------------|-----------------|
| 内容 / 待办 / 事项 / task / title | 待办内容 |
| 负责人 / 处理人 / owner / assignee | 负责人 |
| 截止日期 / 到期 / due / deadline | 截止日期 |
| 状态 / status / state | 状态 |
| 优先级 / priority / level | 优先级 |
| 备注 / 说明 / remark / note | 备注 |

未自动匹配的字段展示给用户确认。

**交互式映射（默认）：**
展示字段映射表，用户可修改：
```
字段映射确认（{.table} 表）：
┌──────────────────┬──────────────────────┐
│ 文件列            │ → Base 字段          │
├──────────────────┼──────────────────────┤
│ 任务内容（自动）   │ 待办内容              │
│ 处理人（自动）     │ 负责人                │
│ 到期日（自动）     │ 截止日期              │
│ 分类（未匹配⚠️）  │ → 选择字段：__       │
└──────────────────┴──────────────────────┘

可选字段：待办内容 | 负责人 | 截止日期 | 状态 | 优先级 | 来源 | 备注 | [跳过]
```

### 第3步：数据验证与去重

1. **必填字段检查**：待办内容不能为空
2. **日期格式检查**：自动识别常见格式（YYYY-MM-DD / YYYY/MM/DD / MM-DD 等）
3. **状态值检查**：与 Base 选项比对，不匹配的标注 ⚠️
4. **负责人解析**：按 CLAUDE.md 公共「成员名称解析」规则，将姓名转为 openId
5. **去重检查**：基于待办内容文本相似度（>85% 视为重复）

### 第4步：预览确认

```
📥 导入预览 — {project_name} · {table}
═══════════════════════════════
总计 {N} 条 | 新记录 {M} 条 | 疑似重复 {D} 条 | 错误 {E} 条

── 新记录（前 5 条预览）──
┌────┬─────────────────┬──────────┬──────────┐
│ #  │ 待办内容          │ 负责人   │ 截止日期  │
├────┼─────────────────┼──────────┼──────────┤
│ 1  │ 完成UI评审        │ 李四     │ 06-20    │
│ 2  │ 提交测试报告      │ 张三     │ 06-18    │
└────┴─────────────────┴──────────┴──────────┘

── 疑似重复（{D} 条）──
  ⚠️ 行 5 "完成UI评审" 与 Base 已有记录 TODO-003 相似度 92%
  ⚠️ 行 12 "提交测试报告" 与行 2 内容高度相似

── 格式问题（{E} 条）──
  ❌ 行 8：待办内容为空
  ⚠️ 行 15：日期格式无法解析 "下周一"，已留空

确认导入？[全部写入] [仅导入新记录] [逐条确认] [取消]
```

### 第5步：写入 Base

- `全部写入`：写入所有新记录（含疑似重复）
- `仅导入新记录`：跳过疑似重复项
- `逐条确认`：逐条展示让用户选择
- 每批最多 200 条，超出的分批串行写入
- 写入失败遵循公共错误重试策略（3 次 ± 退避）

## 异常处理

| 场景 | 处理 |
|------|------|
| 文件不存在 | 提示路径检查 |
| 文件编码无法识别 | 尝试 UTF-8 / GBK，均失败则提示手动转换 |
| 文件超过 10MB | 提示文件过大，建议分批 |
| Base 写入 1254104 | 自动降批，200→100→50 重试 |
| 负责人解析失败 | 字段留空，输出中标注 ⚠️ |
