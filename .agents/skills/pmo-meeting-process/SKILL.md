# pmo-meeting-process — 会议处理

## 执行方式

```bash
# 方式A：飞书妙记
claude pmo-meeting-process --minutes <妙记链接或ID>

# 方式B：外部转写文件
claude pmo-meeting-process --file <本地文件路径>

# 方式C：直接粘贴/输入会议记录文本
claude pmo-meeting-process --text

# 手动指定主题/日期（可选）
claude pmo-meeting-process --minutes <链接> --topic "Sprint评审" --date 2026-06-09
```

## 前置条件

1. 已通过 `pmo-use` 设置当前项目（或已有 SMART_PMO_CURRENT 环境变量）
2. 对于妙记方式：用户有妙记访问权限
3. 对于文件方式：文件存在且格式正确

## 执行流程

### 第1步：获取当前项目配置

```python
# 伪代码逻辑
config = get_current_project_config()
# 从 ~/.smart-pmo/registry/ 读取
```

如果未找到配置，提示：`请先执行 pmo-use <项目名> 设置当前项目`

### 第2步：读取会议内容

**输入源A — 飞书妙记：**
- 通过 `lark-minutes` 获取会议转写内容
- 获取会议元信息：主题、时间、参会人

**输入源B — 外部转写文件：**
- 读取本地文件（支持 .txt / .docx / .md）
- 自动检测编码 UTF-8 / GBK
- 大小限制 ≤ 10MB
- AI 从内容中识别会议主题、日期、参会人

**输入源C — 直接输入文本（--text）：**
- 提示用户粘贴或输入会议记录内容
- 以空行+`EOF`（或用户表示输入完毕）作为结束标记
- 后续处理与输入源B相同
- 记录方式字段写入"手动记录"

### 第3步：AI 提取结构化信息

分析转写内容，提取以下信息：

**会议元信息：**
- 会议主题（参数指定或 AI 识别）
- 会议日期（参数指定或 AI 识别）
- 参会人（妙记元数据或从对话推断）

**讨论要点：**
- 分段总结，每段 1-2 句
- 按讨论顺序排列

**关键决策：**
- 从对话中识别决策点
- 记录决策内容

**待办事项（重点）：**
从转写内容中通过语义模式识别待办：

| 模式 | 示例 | 提取 |
|------|------|------|
| "XX需要/要负责/会处理YY" | "李四负责接口联调" | 内容=接口联调，负责人=李四 |
| "我来做YY" | "我来做演示环境" | 内容=准备演示环境，负责人=说话者 |
| "YY要在DDL前完成" | "周五前要完成方案评审" | 截止日期=本周五 |
| "交给XX" | "这个交给王五处理" | 负责人=王五 |

对于未明确的截止时间，基于当前日期（`currentDate`）动态计算：
- "尽快" / "ASAP" → currentDate + 3天
- "今天" → currentDate
- "明天" → currentDate + 1天
- "这周五" → 本周五日期（若今天已是周五则为下周五）
- "下周X" → 下一周对应的星期X
- "月底" → 当月最后一天
- "下个月底" → 下月最后一天
- 未提及 → **截止日期留空**，确认界面标注 ⚠️ 截止日期待确认

**成员名称解析（负责人字段）— 必须在展示确认前完成：**

识别到姓名后，**写入 Base 前必须完成以下验证**，按顺序执行：
1. 在项目配置 `team.members` 列表中按 `name` 精确匹配 → 获得 openId
2. 精确匹配失败 → 尝试 `team.members` 中的模糊匹配（去掉姓/名前缀）
3. 仍匹配不到 → 通过 `lark-contact` 搜索飞书通讯录（按姓名关键词）
4. 搜索返回多个候选 → 在确认界面列出候选，让用户选择
5. 搜索无结果 → 负责人字段**留空**，确认界面标注 `⚠️ @{姓名} 未匹配，请手动指定`

**未匹配的负责人不阻塞写入流程**，但确认界面需明确展示，用户可以在确认前手动修改。

### 第4步：展示结果给用户确认

按以下格式展示：

```
📋 会议处理结果 · 请确认
──────────────────────────
会议主题: {主题}
会议日期: {日期}
参会人: {列表}

── 讨论要点 ──
1. {要点1}
2. {要点2}

── 关键决策 ──
• {决策1}
• {决策2}

── 待办事项 ──
□ {内容} | @{负责人} | 截止: {日期}
□ {内容} | @{负责人} | 截止: ⚠️ 待确认（未识别到截止时间）
□ {内容} | @??? | 截止: {日期}  ← ⚠️ 负责人未匹配，请确认

[确认写入] [修改后确认] [取消]
```

等待用户选择后进入下一步。

### 第5步：写入数据

写入顺序严格按以下步骤执行，以确保双向关联的完整性：

**① 生成会议纪要文档：**
- 使用模板创建结构化会议纪要
- 通过 `lark-doc` 创建飞书文档，获取 `doc_url`
- 通过 `lark-wiki` 归档到 `01-会议纪要/{YYYY-MM-DD}-{主题}`
- 命名冲突处理：若同名文档已存在，追加序号（如 `2026-06-09-Sprint评审-2`）

**② 写入 Base 会议记录索引表（先写，获取 meeting_record_id）：**
通过 `lark-base` 写入一条记录：
- 会议主题、日期、开始/结束时间、参会人、记录方式
- 来源文件（妙记链接或本地路径）
- 纪要文档链接 = 步骤①得到的 `doc_url`
- 讨论要点摘要、关键决策
- 产出待办 = 暂留空（步骤③创建待办后回填）

→ 获取会议记录的 `meeting_record_id`

**③ 写入 Base 待办事项表（后写，直接填入所属会议）：**
对每个待办，通过 `lark-base` 写入，**同时填入"所属会议"字段 = `meeting_record_id`**：
- 待办内容、负责人、截止日期、状态=待处理、优先级=P2-一般
- 来源="会议:{会议主题}"
- 所属会议 = `meeting_record_id`（从步骤②获取，一次性写入，无需回填）

→ 收集所有写入成功的待办 record_id 列表（`todo_record_ids`）

去重：写入前通过 `lark-base` 查询同来源+相似内容的已有待办（按待办内容关键词匹配），跳过重复项。

**④ 回填会议索引的"产出待办"字段：**
对步骤②写入的会议索引记录，通过 `lark-base update_record` 将"产出待办"字段更新为 `todo_record_ids`（多值关联）。

> ⚠️ 注意：步骤④是"会议索引 ← 待办"方向的回填，即使失败也不影响待办数据的完整性（待办已正确关联到会议）。失败时提示"会议索引的产出待办关联未完成，但不影响待办数据"。

**失败处理策略：**

| 失败点 | 处理方式 |
|--------|---------|
| ①文档归档失败 | 跳过①，继续写②③，doc_url 留空，提示用户手动归档 |
| ②会议索引写入失败 | 提示"会议索引未写入，待办将写入但无会议关联"，待办的所属会议字段留空 |
| ③待办写入失败（部分） | 记录失败项，继续写成功的；④仍使用成功写入的 record_id |
| ④回填产出待办失败 | 提示"会议索引的产出待办关联未完成"，不影响已写入数据 |

## 异常处理

| 场景 | 处理 |
|------|------|
| 无当前项目 | 提示运行 pmo-use |
| 妙记无权限 | 提示使用 --file 方式 |
| 文件格式不支持 | 提示支持的格式 |
| 文件 >10MB | 建议拆分 |
| 未提取到待办 | 提示并询问是否仍写入纪要 |
| Base 写入失败 | 纪要继续归档，待办暂存到 `~/.smart-pmo/.pending_backfill/{project_id}.json` |

### 断点续传机制

步骤④（回填会议索引的产出待办）失败时，将待回填的信息写入本地文件：

```
~/.smart-pmo/.pending_backfill/{project_id}.json
{
  "meeting_record_id": "rec_xxx",
  "todo_record_ids": ["rec_yyy", "rec_zzz"],
  "failed_at": "2026-06-11T15:30:00"
}
```

- 任何 `pmo-*` Skill 执行时，先检查 `~/.smart-pmo/.pending_backfill/` 是否有待处理项
- 若存在 → 自动尝试回填，成功后删除该文件
- 多次失败 → 提示用户 "存在未完成的会议索引关联，请手动检查"
