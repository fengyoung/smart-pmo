# pmo-meeting-process — 会议处理

## 执行方式

```bash
# 方式A：飞书妙记
claude pmo-meeting-process --minutes <妙记链接或ID>

# 方式B：外部转写文件（文件名自动解析日期和主题）
#   文件名格式：YYYYMMDD-{主题}.txt 或 YYYYMMDD-{类型}-{主题}.txt
#   例如：20260410-XRay安卓150款机型推进计划.txt
claude pmo-meeting-process --file <本地文件路径>

# 方式C：直接粘贴/输入会议记录文本
claude pmo-meeting-process --text

# 手动指定主题/日期（可选）
claude pmo-meeting-process --minutes <链接> --topic "Sprint评审" --date 2026-06-09

# 模式选择（可选，组合使用）
claude pmo-meeting-process --file <路径> --doc-only         # 仅生成纪要文档+归档，不写Base
claude pmo-meeting-process --file <路径> --todos-only       # 仅提取待办写入Base，不生成纪要
claude pmo-meeting-process --index-only <会议主题> <日期>   # 仅为已有待办创建会议索引+双向关联
claude pmo-meeting-process --file <路径> --sub-project <项目名>  # 从跨项目周会中提取指定项目专题

# 新增：跳过交互确认 / 生成本地文件（适用于批量/离线场景）
claude pmo-meeting-process --file <路径> --no-confirm       # 跳过交互确认，直接写入
claude pmo-meeting-process --file <路径> --local            # 生成本地 .md 文件，不创建飞书文档
claude pmo-meeting-process --file <路径> --no-confirm --local  # 批量模式：静默生成本地 .md
claude pmo-meeting-process --file <路径> --local --asr-correction ./ASR校正表.md  # 本地模式指定 ASR 校正表
```

## 前置条件

1. 已通过 `pmo-use` 设置当前项目（或已有 SMART_PMO_CURRENT 环境变量）
   - **`--local` 模式例外**：本地 .md 输出不需要项目配置；如需 ASR 校正请通过 `--asr-correction <文件>` 参数指定
2. 对于妙记方式：用户有妙记访问权限
3. 对于文件方式：文件存在且格式正确

## 执行流程

### 第1步：获取当前项目配置

```python
# 伪代码逻辑
config = get_current_project_config()
# 从 ~/.smart-pmo/registry/ 读取
```

- 如果未找到配置且 **不是 `--local` 模式**，提示：`请先执行 pmo-use <项目名> 设置当前项目`，中断执行
- 如果未找到配置且 **是 `--local` 模式**，提示：`⚠️ 未找到项目配置，将跳过成员映射等上下文增强，继续执行（降级为纯文本提取）`。ASR 校正表仍可通过 `--asr-correction <文件>` 参数加载

### 第2步：读取会议内容

**输入源A — 飞书妙记：**
- 通过 `lark-minutes` 获取会议转写内容
- 获取会议元信息：主题、时间、参会人

**输入源B — 外部转写文件：**
- 读取本地文件（支持 .txt / .docx / .md）
- 自动检测编码 UTF-8 / GBK
- 大小限制 ≤ 10MB
- **文件名自动解析（新增）：** 若文件名符合 `YYYYMMDD-{主题}` 或 `YYYYMMDD-{类型}-{主题}` 格式，自动提取日期和主题作为默认值
  - 例如 `20260410-XRay安卓150款机型推进计划.txt` → 日期=2026-04-10, 主题=「XRay安卓150款机型推进计划」
  - 若 `--date` 或 `--topic` 参数已指定，以参数为准
  - 解析结果在确认界面展示：`📝 从文件名解析到：日期={date}, 主题={topic}`
- AI 从内容中识别会议主题、日期、参会人（作为文件名解析的补充/兜底）

**输入源C — 直接输入文本（--text）：**
- 提示用户粘贴或输入会议记录内容
- 以空行+`EOF`（或用户表示输入完毕）作为结束标记
- 后续处理与输入源B相同
- 记录方式字段写入"手动记录"

### 🔄 第2.5步：加载 ASR 校正表 + Wiki 上下文（新增）

**执行 AI 提取前，先加载以下辅助上下文以提升准确性：**

**① ASR 校正表加载：**
- 非 `--local` 模式：通过 `lark-wiki` 检查项目知识库 `05-项目资料/ASR校正表.md` 是否存在
- `--local` 模式：检查是否指定 `--asr-correction <文件>` 参数，若指定 → 读取本地校正表文件；未指定 → 跳过
- 若存在 → 读取内容作为 LLM prompt 的校正上下文
- 若不存在/未指定 → 跳过，不影响主线流程

**② 历史会议参考加载：**
- `--local` 模式：跳过（无本地 Base 访问）
- 非 `--local` 模式：通过 `lark-base` 查询当前项目「会议记录索引」表，取最近 2 条记录（按会议日期倒序）
- 提取其「讨论要点摘要」作为近会议上下文参考
- 若无历史记录 → 跳过

**③ 项目成员上下文：**
- `--local` 模式：跳过（仅保留文本姓名，不解析 openId）
- 非 `--local` 模式：从项目配置 `team.members` 读取成员列表，组装为姓名+角色映射表
- 用于负责人识别时的第一轮匹配参考

**处理规则：**
- 以上三步均为非阻塞增强，任一加载失败不影响主线流程
- 加载成功时在终端输出：`📖 已加载 ASR校正表（{N}条）、历史会议参考（{N}条）、成员映射（{N}人）`
- 加载的内容注入到 LLM prompt 的「背景上下文」段（详见 Prompt 模板 `references/meeting-prompt-template.md`）

### 第3步：AI 提取结构化信息

分析转写内容，提取以下信息。**调用 LLM 时使用结构化 Prompt 模板（详见 `_shared/meeting-prompt-template.md`），模板包含三段式结构：背景上下文 → 任务说明 → 输出格式。**

**ASR 校正（前置环节）：**
- 将第 2.5 步加载的 ASR 校正表注入 prompt 的「背景上下文」段
- 要求 LLM 先校正转写文本中的人名和术语误识别，再进行信息提取
- 校正结果不单独输出，仅用于提高后续提取准确性

**会议元信息：**
- 会议主题（优先级：`--topic` 参数 > 文件名解析 > AI 识别）
- 会议日期（优先级：`--date` 参数 > 文件名解析 > AI 识别）
- 参会人（妙记元数据或从对话推断，映射到注册成员名）

**讨论要点：**
- 分段总结，每段 1-2 句
- 按讨论顺序排列

**关键决策：**
- 从对话中识别决策点
- 记录决策内容

**待办事项（重点）：**
从转写内容中通过语义模式识别待办：

| 模式 | 示例 | 提取 |
|------|------|------|
| "XX需要/要负责/会处理YY" | "李四负责接口联调" | 内容=接口联调，负责人=李四 |
| "我来做YY" | "我来做演示环境" | 内容=准备演示环境，负责人=说话者 |
| "YY要在DDL前完成" | "周五前要完成方案评审" | 截止日期=本周五 |
| "交给XX" | "这个交给王五处理" | 负责人=王五 |

> 📅 截止时间计算规则详见 [`_shared/date-calc-rules.md`](_shared/date-calc-rules.md)：
> - "尽快" / "ASAP" → currentDate + 3天
> - "今天" → currentDate、"明天" → currentDate + 1天
> - "这周五" → 本周五（若今天已是周五则为下周五）
> - "下周X" → 下一周对应的星期X、"月底" → 当月最后一天
> - 未提及 → **截止日期留空**，确认界面标注 ⚠️ 截止日期待确认

**成员名称解析（负责人字段）：**

- **`--local` 模式**：仅保留文本姓名，不解析 openId，不匹配飞书通讯录。负责人直接以文本形式写入纪要的待办表格。
- **非 `--local` 模式**：必须在展示确认前完成以下验证：
  1. 在项目配置 `team.members` 列表中按 `name` 精确匹配 → 获得 openId 和姓名
  2. 精确匹配失败 → 尝试 `team.members` 中的模糊匹配（去掉姓/名前缀）
  3. 仍匹配不到 → 通过 `lark-contact` 搜索飞书通讯录（按姓名关键词）→ 获取 `open_id` 和 `localized_name`
  4. 搜索返回多个候选 → 在确认界面列出候选，让用户选择
  5. 搜索无结果 → 负责人字段**留空**，确认界面标注 `⚠️ @{姓名} 未匹配，请手动指定`

  **负责人字段写入格式（⚠️ 关键）：**
  写入 Base 时必须同时传 `id` 和 `name`，**仅传 `id` 会触发 `1254066 UserFieldConvFail`**：
  ```json
  {"负责人": [{"id": "ou_xxx", "name": "姓名"}]}
  ```
  多负责人时依次追加到数组中。

  **负责人写入失败降级策略（P1-5）：**
  写入 Base 后，逐条验证负责人字段是否写入成功：
  - 若某条待办的负责人字段写入失败 → 自动将负责人姓名写入**备注字段**（格式：`负责人: {姓名列表}`）
  - 将失败记录追加到 `~/.smart-pmo/.pending_assignee/{project_id}.json`
  - 终端明确提示：`⚠️ @{姓名} 写入负责人失败（API限制），已写入备注列，请在 Base 中手动分配`
  - `pmo-todo-followup` 执行时检查 `.pending_assignee/`，若有待处理项则提示用户

**未匹配的负责人不阻塞写入流程**，但确认界面需明确展示，用户可以在确认前手动修改。

### 第4步：展示结果给用户确认

**如果指定了 `--no-confirm`：**
跳过本步，直接进入第 5 步。根据 `--local`/`--doc-only`/`--todos-only` 等模式标志选择写入路径：
- `--local` → 写入本地 .md 文件（详见第 5 步 `--local` 模式）
- `--doc-only` → 生成飞书纪要文档+归档
- `--todos-only` → 写入待办到 Base
- 无模式标志 → 执行完整流程（全部写入）

**如果未指定 `--no-confirm`：**
按以下格式展示结果，等待用户选择：

```
📋 会议处理结果 · 请确认
──────────────────────────
会议主题: {主题}
会议日期: {日期}
参会人: {列表}

── 讨论要点 ──
1. {要点1}
2. {要点2}

── 关键决策 ──
• {决策1}
• {决策2}

── 待办事项 ──
□ [1] {内容} | @{负责人} | 截止: {日期}
□ [2] {内容} | @{负责人} | 截止: ⚠️ 待确认（未识别到截止时间）
□ [3] {内容} | @??? | 截止: {日期}  ← ⚠️ 负责人未匹配，请确认

[全部写入] [仅写待办] [仅写纪要] [选择写入] [修改] [取消]
```

**确认选项说明（P0-2）：**

| 选项 | 行为 |
|------|------|
| **全部写入** | 执行完整流程：生成纪要→归档知识库→写会议索引→写待办→回填关联 |
| **仅写待办** | 跳过①纪要生成，执行②③④：写会议索引→写待办→回填关联 |
| **仅写纪要** | 只执行①：生成纪要文档→归档知识库，不写 Base |
| **选择写入** | 用户输入待办编号（如 `1,3,5-7`），只写入选中的待办 + 对应的会议索引 |
| **修改** | 用户输入修改内容，调整后重新展示确认界面 |
| **取消** | 放弃本次操作，缓存解析结果到草稿（见下方草稿机制） |

等待用户选择后进入下一步。

### 第5步：写入数据

写入方式根据模式标志决定：

**`--local` 模式（生成本地 .md 文件）：**
跳过所有飞书文档和 Base 操作，直接输出本地 Markdown 文件：
- 使用结构化 Prompt 模板生成的会议纪要正文
- 在附录附上原始转写文本
- 末尾添加签名：`*文档生成日期：{生成日期}*`
- 输出路径规则：
  - 若指定了 `--output <路径>`，写入指定路径
  - 否则与输入文件同目录，同名 `.md` 文件
  - 例如：`20260615-内部讨论.txt` → `20260615-内部讨论.md`
- 输出完成后打印：`✅ 已生成本地纪要：{output_path}`

**非 `--local` 模式（飞书+Base 写入）：**
写入顺序严格按以下步骤执行，以确保双向关联的完整性：

**模式适配：**
- `--todos-only`：只执行②③④，跳过①
- `--doc-only`：只执行①，跳过②③④
- `--index-only`：只执行② + 更新已有待办的"所属会议"字段 + 回填④
- `选择写入`：仅处理用户选中的待办，其余步骤照常
- `--sub-project <项目名>`：在①的纪要文档中标注"提取自 {主项目} 周会，本文档为 {sub-project} 子专题"，②③使用 sub-project 的 Base 配置

**`--sub-project` 边界情况处理：**

| 情况 | 处理规则 |
|------|---------|
| sub-project 项目名未注册 | 提示"项目 {名称} 未注册，请先运行 pmo-init 或 pmo-use"，中断执行 |
| sub-project 与当前项目相同 | 提示"--sub-project 不能与当前项目相同"，中断执行 |
| 成员解析：成员姓名在主项目配置中存在 | 优先使用主项目 `team.members` 解析 openId |
| 成员解析：主项目未匹配 → sub-project 配置 | 尝试 sub-project `team.members` 解析 |
| 成员解析：两个项目均未匹配 | 通过 `lark-contact` 搜索飞书通讯录 |
| 去重检查 | 仅在 sub-project 的待办表中检查，不跨到主项目 |
| 文档归档 | 归档到 sub-project 知识库的 `01-会议纪要/` 目录 |
| 会议索引 | 写入 sub-project 的会议记录索引表 |
| 纪要文档标题格式 | `{YYYYMMDD}-{主项目}周会-{sub-project}专题` |

**① 生成会议纪要文档：**
- 使用模板创建结构化会议纪要
- 通过 `lark-doc` 创建飞书文档，获取 `doc_url`
- 通过 `lark-wiki` 归档到 `01-会议纪要/{YYYYMMDD}-{title}`
- 命名冲突处理：若同名文档已存在，追加序号（如 `20260609-Sprint评审-2`）
- `--sub-project` 时：文档标题加注 `{主项目}周会 — {sub-project}专题`

**② 写入 Base 会议记录索引表（先写，获取 meeting_record_id）：**
通过 `lark-base` 写入一条记录：
- 会议主题、日期、开始/结束时间、参会人、记录方式
- 来源文件（妙记链接或本地路径）
- 纪要文档链接 = 步骤①得到的 `doc_url`（`--todos-only` 模式下可为空，标注 `⚠️ 未生成纪要`）
- 讨论要点摘要、关键决策
- 产出待办 = 暂留空（步骤③创建待办后回填）
- `--sub-project` 时：会议主题前置 `[{sub-project}] `，备注中写明完整会议为 {主项目}周会

→ 获取会议记录的 `meeting_record_id`

**③ 写入 Base 待办事项表（后写，直接填入所属会议）：**
对每个待办，通过 `lark-base` 写入，**同时填入"所属会议"字段 = `meeting_record_id`**：
- 待办内容、负责人、截止日期、状态=待处理、优先级=P2-一般
- 来源="会议:{会议主题}"
- 所属会议 = `meeting_record_id`（从步骤②获取，一次性写入，无需回填）

→ 收集所有写入成功的待办 record_id 列表（`todo_record_ids`）

**去重检查（P2-6）：**
写入前通过 `lark-base` 查询同来源+相似内容的已有待办（按待办内容关键词匹配）：
- 若发现候选重复项 → 在终端展示：
  ```
  ⚠️ 检测到可能重复的待办：
    NO.002 15/16分类模型开发（已有，6/3会议录入）← 待写入: 15/16分类模型标注推进
    NO.005 强推2.0策略决策会（已有，6/2会议录入）← 待写入: 强推2.0策略决策会
  上述 {N} 条跳过写入，其余继续？
  [是] [否，全部写入] [手动选择]
  ```
- 用户确认后跳过重复项写入

**④ 回填会议索引的"产出待办"字段：**
对步骤②写入的会议索引记录，通过 `lark-base update_record` 将"产出待办"字段更新为 `todo_record_ids`（多值关联）。

> ⚠️ 注意：步骤④是"会议索引 ← 待办"方向的回填，即使失败也不影响待办数据的完整性（待办已正确关联到会议）。失败时提示"会议索引的产出待办关联未完成，但不影响待办数据"。

**失败处理策略：**

| 失败点 | 处理方式 |
|--------|---------|
| ①文档归档失败 | 跳过①，继续写②③，doc_url 留空，提示用户手动归档 |
| ②会议索引写入失败 | 提示"会议索引未写入，待办将写入但无会议关联"，待办的所属会议字段留空 |
| ②成功但③全部失败 | **孤立会议记录**：将 meeting_record_id + 原始待办内容保存到 `~/.smart-pmo/.pending_orphan_meeting/{project_id}.json`，提示用户执行 `--index-only` 补录 |
| ③待办写入失败（部分） | 记录失败项，继续写成功的；④仍使用成功写入的 record_id |
| ③待办负责人字段写入失败 | 自动降级：将负责人姓名写入备注字段，记录到 `.pending_assignee/` |
| ④回填产出待办失败 | 记录到 `.pending_backfill/`，提示"会议索引的产出待办关联未完成"；见 CLAUDE.md 人工介入出口 |

## 异常处理

| 场景 | 处理 |
|------|------|
| 无当前项目（非 `--local`） | 提示运行 pmo-use |
| 无当前项目（`--local`） | 降级为纯文本提取，跳过上下文增强 |
| 妙记无权限 | 提示使用 --file 方式 |
| 文件格式不支持 | 提示支持的格式 |
| 文件 >10MB | 建议拆分 |
| 未提取到待办 | 提示并询问是否仍写入纪要 |
| Base 写入失败 | 纪要继续归档，待办暂存到 `~/.smart-pmo/.pending_backfill/{project_id}.json` |
| 负责人写入失败 | 自动降级写备注字段，记录到 `.pending_assignee/`（P1-5） |
| 全部Base查询失败 | 回退到仅生成纪要模式，提示用户稍后手动补录 |

### 草稿缓存机制（P1-4）

用户选择「取消」或异常中断时，自动将解析结果缓存：

```
~/.smart-pmo/.draft/{project_id}_{date}_{topic_slug}.json
{
  "source_file": "/path/to/transcript.txt",
  "topic": "会议主题",
  "date": "2026-06-03",
  "participants": ["张三", "李四"],
  "discussion_points": [...],
  "decisions": [...],
  "todos": [...],
  "cached_at": "2026-06-11T15:30:00"
}
```

**恢复机制：**
- 下次执行 `pmo-meeting-process --file <同文件>` 时自动检测对应草稿
- 提示：`📝 检测到上次缓存的解析结果（{cached_at}），是否直接使用？ [是] [否，重新分析]`
- 选择「是」→ 跳过第3步（AI提取），直接展示第4步确认界面
- 确认写入成功后自动清理草稿文件
- 草稿超过 7 天未使用 → 下次检测到时提示并建议删除

### 独立创建会议索引（P1-3）

`--index-only` 模式用于为已存在的待办建立会议索引和双向关联：

**输入：**
```
claude pmo-meeting-process --index-only "<会议主题>" "<会议日期>"
```

**交互流程：**
1. 提示用户输入或选择待办 record_id 列表（可通过 Base 链接或 ID 指定）
2. 提示输入纪要文档链接（可选，已归档则填入）
3. 写入会议索引记录 → 获取 meeting_record_id
4. 批量更新所选待办的"所属会议"字段 → meeting_record_id
5. 回填会议索引的"产出待办"字段 → 待办 record_id 列表

**使用场景：**
- 分批写入待办后，统一建立会议关联
- `--todos-only` 后补建会议索引
- 跨项目周会后追溯建立索引
