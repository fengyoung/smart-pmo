# pmo-todo-from-chat — 群消息提取待办 Skill 设计

> P0 优先级 | 从群聊对话中自动提取待办

---

## 功能概述

通过 `lark-im` 读取项目群聊中未被处理的消息，AI 分析提取潜在待办事项，经用户确认后写入 Base「待办事项」表。

---

## 调用方式

```bash
claude pmo-todo-from-chat
```

---

## 执行流程

```
用户输入: claude pmo-todo-from-chat
         │
         ▼
① 获取项目配置中的 lastReadMessageId
   └→ 如果为空 → 读取最近 7 天的消息（最多 100 条）
   └→ 如果不为空 → 从 lastReadMessageId 下一条开始读取
         │
         ▼
② 通过 lark-im 读取群聊消息
   ├→ API: list_message(chat_id, page_size=100, start_time=lastReadTime)
   ├→ 读取上限：最多 100 条
   ├→ 过滤规则：
   │    ├→ 排除系统消息（type=system）
   │    ├→ 排除 Bot 自己发的消息（避免自循环）
   │    └→ 排除 @Bot 的指令类消息（/todo 等）
   └→ 保留原始消息 ID 用于排重
         │
         ▼
③ AI 分析消息内容
   ├→ 语义分析识别"待办意图"
   ├→ 提取内容：待办描述、负责人、截止时间
   ├→ 去重判断：与 Base 已有待办对比（语义相似度 >80% 标记重复）
   └→ 过滤：明显非待办的对话（日常闲聊、表情包等）
         │
         ▼
④ 展示提取结果
   ┌─────────────────────────────────────────┐
   │ 📋 从群消息中发现以下潜在待办             │
   │                                         │
   │ 分析范围: 2026-06-08 18:00 ~ 06-09 10:00│
   │ 分析消息: 42 条                           │
   │ 提取待办: 3 条（1 条已在 Base 中）        │
   │                                         │
   │ □ [新] V2.0接口联调@李四 6/16前完成      │
   │   → 来源: 李四 06-09 09:32               │
   │                                         │
   │ □ [新] 准备演示环境@张三                  │
   │   → 来源: 王五 06-09 09:45               │
   │   ⚠️ 截止时间未指定                       │
   │                                         │
   │ ⚠️ [已存在] 更新项目计划@张三 6/11        │
   │   → 已在待办列表中，跳过                  │
   │                                         │
   │ [全部写入] [选择写入] [取消]              │
   └─────────────────────────────────────────┘
         │
         ▼
⑤ 用户确认后：
   ├→ 通过 lark-base 写入「待办事项」表
   │   ├→ 来源 = "群聊"
   │   ├→ 来源消息ID = 原始消息 ID
   │   └→ 状态 = "待处理"
   └→ 更新配置中的 lastReadMessageId 和 lastReadTime
         │
         ▼
⑥ 推送反馈到项目群
   └→ "已从群消息提取 2 条待办并录入 Base"
```

---

## 去重机制详细设计

### 层级1：消息级别去重

```
条件：来源消息ID 已在 Base 中
策略：跳过该消息
实现：Base「待办事项」表的「来源消息ID」字段记录原始消息 ID
      读取新消息时，与 Base 中已有 ID 比对
```

### 层级2：内容级别去重

```
条件：来源消息ID 不同，但待办内容语义相似度 >80%
策略：标记为重复，提示用户该待办可能已存在
匹配维度：
  - 内容文本的语义相似度
  - 负责人是否相同
  - 是否为同一项目
```

### 层级3：用户确认兜底

```
最终由用户判断是否写入，给予"全部写入" / "选择写入" / "取消" 三个选项
```

## 离线场景说明

| 场景 | 行为 |
|------|------|
| Claude 不常开 | 每次打开执行一次即可捕获自上次处理以来的所有新待办 |
| 消息量大 | 每次最多读 100 条，可多次执行逐步处理 |
| 消息已有 Base 记录 | 内容去重机制跳过，不会重复录入 |

---

## 配置变更

每次成功执行后更新 `pmo-config.json`：

```json
{
  "chat": {
    "lastReadMessageId": "om_prev_last_id",
    "lastReadTime": "2026-06-09T10:00:00"
  }
}
```

---

## 异常处理

| 场景 | 处理方式 |
|------|---------|
| 无未处理消息 | 提示"自上次处理以来无新消息" |
| 无法连接飞书 API | 提示检查网络，稍后重试 |
| 消息量超大（群活跃）| 提示"发现较多未处理消息，建议分多次执行" |
| AI 提取未发现待办 | 提示"未识别到新待办，如需放宽识别标准请调整" |
