## P0 — 关键问题（必须修复）

### P0-1. pmo-archive 缺少交互式确认

**违反原则**：CLAUDE.md 明确声明「交互优先 — 关键操作需用户确认后再执行」，但 pmo-archive 在执行文件上传到知识库前，**没有任何用户确认步骤**，直接执行写入操作。

**影响**：用户可能误归档文件到错误目录，且无法撤销。

**修复建议**：
- 在「第 2 步：上传到知识库」前增加确认提示，列出：文件名、目标目录、文件大小
- 参考 pmo-changelog 的「草稿预览 + 4 选项」交互模式

### P0-2. REQUIREMENTS.md 路线图过期

**现状**：
- 阶段五（P3）标记为「📋 规划中」，但实际 v1.6.0 已全部完成
- Skill 列表（第 8 节）缺少 6 个 v1.6.0 Skill：`pmo-burn-down`、`pmo-changelog`、`pmo-retro`、`pmo-health`、`pmo-standup`、`pmo-weekly-digest`
- 最后更新日期仍为 `2026-06-12`

**影响**：新用户/贡献者看到的需求清单与实际项目状态严重不符。

**修复建议**：
- 更新路线图，标记阶段五（P3，v1.6.0）为 ✅ 已完成
- 补充缺失的 6 个 Skill 到第 8 节列表
- 添加阶段六（P4 规划）占位
- 更新日期为 `2026-06-16`

### P0-3. 用户级 settings.json 包含 API Token

**现状**：`~/.claude/settings.json` 包含明文 `ANTHROPIC_AUTH_TOKEN`（`sk_7ac...`）。

**风险**：虽然不在仓库中，但任何能访问用户系统的进程都可以读取该 token。

**修复建议**：
- 将 token 迁移到环境变量或 macOS Keychain
- 参考 CLAUDE.md 中已有的 Keychain 模式：`Bash(security find-generic-password *)` 已出现在 settings.local.json 的 allow 列表中

## P1 — 重要问题（应该修复）

### P1-1. pmo-meeting-process：Prompt 模板路径引用不一致

**现状**：
- 第 114 行：`references/meeting-prompt-template.md` ✅（实际文件位置）
- 第 118 行：`_shared/meeting-prompt-template.md` ❌（路径错误）

**影响**：LLM 按 `_shared/` 路径查找文件会失败，导致结构化 Prompt 模板无法加载。

**修复**：将第 118 行的 `_shared/meeting-prompt-template.md` 改为 `references/meeting-prompt-template.md`。

### P1-2. pmo-todo-followup 完全缺失待处理队列检查

**现状**：SKILL.md 中没有任何对 `_shared/pending-queue-check.md` 的引用，也没有内联版本的待处理队列检查逻辑。

**影响**：
- 其他 Skill（如 pmo-meeting-process）写入失败的待办回填/孤立会议，在 pmo-todo-followup 执行时不会被提示
- `.pending_assignee/` 中的负责人写入失败记录不会被检查和提示

**修复**：在「前置条件」后增加待处理队列检查章节，引用 `_shared/pending-queue-check.md`。

### P1-3. pmo-todo-from-chat 日期计算内联 + 配置加载不规范

**现状**：
- 日期计算规则完全内联在 SKILL.md 中，而非引用 `_shared/date-calc-rules.md`
- 配置加载使用内联伪代码 `get_current_project_config()`，未展示标准的 6 步加载序列
- 缺少显式的配置完整性校验

**影响**：日期计算逻辑与共享模块可能不同步；配置加载不符合统一约定。

**修复**：
- 将日期计算替换为 `_shared/date-calc-rules.md` 引用
- 补全标准的配置加载 + 校验章节

### P1-4. pmo-archive 缺失配置加载/校验/重试三个公共模式

**现状**：pmo-archive 是唯一同时缺失以下三个 CLAUDE.md 公共模式的 Skill：
- 无标准配置加载序列（仅「前置条件」一句模糊描述）
- 无配置完整性校验
- 无错误重试策略（异常表只有基础 4 行）

**修复**：参照 pmo-burn-down 的结构，补充完整的三个公共模式章节。

### P1-5. pmo-init 的 depends_on 缺少 lark-contact

**现状**：SKILL.md frontmatter 中 `depends_on` 仅列出 `lark-wiki`、`lark-base`、`lark-im`，但「成员搜索确认」步骤使用了 `lark-contact` 进行飞书通讯录搜索。

**修复**：在 depends_on 中增加 `lark-contact`。

### P1-6. tests/skill-checklist.md 缺少 6 个 v1.6.0 Skill 测试

**现状**：测试 checklist 仅覆盖到 pmo-meeting-prep，缺少：
- pmo-changelog、pmo-burn-down、pmo-retro、pmo-health、pmo-standup、pmo-weekly-digest

**修复**：为每个新 Skill 补充 Happy Path + Edge Cases + Failure Recovery 测试场景。

### P1-7. .claude/settings.local.json 权限过于宽松

**现状**：
- `ssh *` — 允许任意 SSH 连接
- `scp *` — 允许任意文件传输
- `curl *` — 允许任意 HTTP 请求
- `cat` — 允许读取任意文件
- `Bash(python3 -c ' *)` — 允许任意 Python 代码执行

**风险**：如果某个 Skill 的 LLM 输出被注入恶意命令，攻击面过大。

**修复建议**：
- `ssh *` → 限定到已知主机：`ssh root@101.42.1.145`
- `scp *` → 限定源/目标路径
- `curl *` → 限定域名白名单
- `cat` → 限定路径前缀
- `python3 -c ' *)` → 限定到特定脚本路径
